## Build-app course mode

Review the script as a **build-along** episode about a **real application**.

### Balance check (mandatory)

Respect active Narrative balance (by topic headers Hook/Build/Demo/Recap — no clocks):

| Mode | Practice | Theory |
|------|----------|--------|
| practice-first | ~65–75% | ~25–35% |
| balanced | ~50% | ~50% |
| theory-first | ~35% | ~65% |

Flag when Recap/Theory words dominate a practice-first episode, or when theory-first lacks a real file walkthrough + hero demo.

### Length

Respect the episode word band. Prefer **CUT for budget** recommendations that remove repetition/theory padding — always pair "add" with "remove".

### Duplicate mechanism (mandatory)

Flag when the **same algorithm/mechanism** is fully explained twice (e.g. RRF math in Theory **and** again in Build).  
Recommend: keep the Build explanation; cut Theory down to a 1–2 sentence name+why.

Flag Theory/Recap blocks that contain **formulas or rank arithmetic** — those belong next to code.

### Thin close

If `## Recap` (or closing block) re-lists the whole pipeline, mark **CUT for budget**: keep 3–5 sentences (built → showed on camera → abstract bridge). No second BM25/vector/RRF/rerank lecture.

### Series language

Flag and cut: "next video", "next episode", "video N", "episode N".  
Allow only abstract series bridges.

### Educational priorities for build-app

1. Hook states what will be built and what success **looks like** on camera.
2. Theory refreshes context for the code — it must not replace the walkthrough.
3. File walkthrough teaches role → outcome → key imports/types/methods.
4. Demo must show the implemented result (e.g. chunk text/overlap, rank move), not only a successful exit code.
5. Dedup against Prior coverage / Concepts introduced.

Keep edits concrete (which paragraphs to cut/merge, which file or demo to expand).
